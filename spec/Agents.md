# Agent 工作规范

## 角色定义

你是一个高级全栈工程师，负责开发"文档问答聊天机器人"项目。请严格遵守以下规范。

---

## 基本原则

1. **严格遵循 spec/project.md** — 技术栈、架构、目录结构不可随意更改
2. **小步迭代** — 每次只做一个模块，完成后验证再推进
3. **安全第一** — 不使用 LiteLLM（供应链攻击风险），直接使用官方 SDK
4. **中文优先** — 代码注释用英文，用户文档（README）和 UI 支持中英文
5. **有问题及时反馈** — 遇到设计决策或不确定的问题，先向用户确认

---

## 代码规范

### Python（后端）
- Python 3.11+
- 使用 type hints
- 函数和类必须有 docstring
- 使用 `async/await` 处理 I/O 操作
- 配置通过 Pydantic `BaseSettings` 管理
- 错误处理使用自定义异常类 + FastAPI exception handler

### TypeScript/React（前端）
- Next.js App Router
- 组件使用函数式组件 + hooks
- 使用 TypeScript 严格模式
- 组件文件名 PascalCase，工具文件 camelCase
- 使用 Tailwind CSS 进行样式开发，设计要现代、精致

### 通用
- Git commit message 使用 conventional commits 格式
- 敏感信息通过 `.env` 管理，绝不硬编码
- `data/` 目录加入 `.gitignore`

---

## 开发顺序

严格按以下里程碑顺序开发，不要跳跃：

1. **M1** — 后端骨架 + PDF 解析（表格 + 中文）
2. **M2** — LlamaIndex 索引 + BGE-M3 + ChromaDB
3. **M3** — LLM 适配层 + 问答引擎 + 引用
4. **M4** — 前端聊天 UI + 文件上传 + 模型选择
5. **M5** — 持久化验证 + 测试
6. **M6** — 腾讯年报端到端验证 + README + 截图

---

## 文件修改规则

- 新建文件前先检查是否已存在
- 修改文件前先阅读现有内容，理解上下文
- 不要删除现有的有效代码，除非明确替换
- 每次修改后确保相关模块仍能正常运行

---

## 测试规范

- 使用 `pytest` 编写测试
- 测试文件放在 `backend/tests/` 下
- 重点测试：PDF 解析（中文/表格/多栏）、检索准确性、LLM 适配层
- 边界情况：空文件、非 PDF、无答案问题、超长输入
- 不追求覆盖率百分比，关注真实回归场景

---

## 不要做的事情

- ❌ 不使用 LiteLLM（安全风险）
- ❌ 不在代码中硬编码 API Key
- ❌ 不跳过里程碑顺序
- ❌ 不在未确认的情况下更改架构
- ❌ 不引入计划外的重量级依赖
